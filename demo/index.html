Matriz de evidencia

La demo debe dejar claro que el producto resiste cambios y no solo reproduce estados felices.

Validacion antes de fallar

Email y contexto insuficiente bloquean la simulacion sin romper el estado actual.

Persistencia ligera

El draft vuelve tras un reload y evita perder contexto durante sesiones de evaluacion.

Errores recuperables

Timeout y bloqueo por aprobacion muestran siguientes pasos claros en lugar de callejones sin salida.

Contrato publico

La integracion escucha eventos `nds-*` desde el host y decide el estado visible con su propia logica.

Como leer esta demo ahora

Primero valida comportamiento; despues entra al catalogo para revisar la API de cada pieza.

  • Empieza por `Workspace rollout` si quieres evaluar formularios, estado y retry.
  • Abre `Ops console` para ver feedback y dialogos en una operacion con riesgo.
  • Usa `Host integration lab` para decidir cuando necesitas `shadow`, `light` o un mix.
  • Consulta `DOM modes` si quieres la comparativa corta y tecnica.
  • Entra al catalogo cuando necesites atributos, eventos y snippets concretos.

Construye una vez. Demuestra fiabilidad en varios contextos.

La demo ya no es solo una vitrina de componentes. Es una prueba de que el mismo paquete aguanta estados, integraciones y decisiones reales de producto.
